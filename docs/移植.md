# 哨兵导航与决策移植说明

## 1. 当前结论

当前工程已经完成以下迁移结果：

- `rmul_2026.xml` 是新的主决策树
- 决策执行方式已经切换为 `NavigateThroughPoses`
- 默认轻量仿真不再依赖裁判系统时间、血量、RFID
- loopback 默认走 `simulation` 输入源
- 同时保留了可选的“手动裁判仿真输入”模式，便于测试比赛状态、血量、时间分支

这意味着你现在的项目已经不是 `goal_pose` 方案，而是：

- 决策层：`pb2025_sentry_behavior`
- 导航执行层：`Nav2 NavigateThroughPoses`
- 轻量仿真层：`nav2_loopback_sim`

## 3. 现在的决策结构

### 3.1 顶层输入源

`rmul_2026` 现在支持两种输入源：

- `decision.input_source: simulation`
- `decision.input_source: referee`

行为树会先判断输入源，再选择对应分支：

- `simulation` 分支只看 `decision/sim_mode`
- `referee` 分支继续看比赛状态、剩余时间、当前血量

### 3.2 simulation 分支

支持四种仿真模式：

- `patrol`
- `anchor`
- `retreat`
- `safe`

对应关系：

- `patrol`：多点巡逻
- `anchor`：固定锚点驻守
- `retreat`：退防点
- `safe`：安全点

### 3.3 referee 分支

优先级从高到低：

1. 危险血量 `critical` -> 安全点
2. 低血量 `low` -> 退防点
3. 关键时间 `critical` 且血量正常 -> 关键时间固定点
4. 正常血量 -> 巡逻

## 4. 现在为什么更平滑

这次移植里，决策输出已经不是反复发布单个 `/goal_pose`，而是直接调用 `NavigateThroughPoses`：

- 巡逻路径可以一次性交给 Nav2 连续执行
- 分支不变时，不会每个 tick 重复取消并重发同一路径
- 分支切换时，会主动取消旧路径并派发新路径

同时增加了参数：

- `decision.decision_config.patrol_preview_points`

这个参数的作用是：

- 决定一次送给 `NavigateThroughPoses` 的巡逻预览点数量
- 数值越大，巡逻点足够多时越连续
- 不需要重新编译，只要改 YAML

注意：

- 如果你的 `patrol_indices` 只有 2 个点，那么 `patrol_preview_points` 调再大，提升也有限
- 真正想更丝滑，应该同时增加巡逻点数量和路径密度

运行中临时调这个参数也可以：

```bash
ros2 param set /pb2025_sentry_behavior_server decision.decision_config.patrol_preview_points 3
```

如果你是改 YAML，则重新启动 launch 即可，不需要重新编译。

## 6. 轻量 loopback 默认运行方式

默认启动：

```bash
source install/setup.bash
ros2 launch pb2025_sentry_bringup loopback_decision_sim.launch.py
```

如果你手动指定，也推荐写成：

```bash
ros2 launch pb2025_sentry_bringup loopback_decision_sim.launch.py use_rviz:=true
```

默认行为：

- 使用 `sentry_behavior_loopback.yaml`
- `decision.input_source=simulation`
- `fake_decision_sim_inputs.py` 发布 `decision/sim_mode`
- 行为树执行 `rmul_2026`

运行中切换模式：

```bash
ros2 param set /fake_decision_sim_inputs decision_mode patrol
ros2 param set /fake_decision_sim_inputs decision_mode anchor
ros2 param set /fake_decision_sim_inputs decision_mode retreat
ros2 param set /fake_decision_sim_inputs decision_mode safe
```

如果想完全自己手动发仿真模式 topic：

```bash
ros2 launch pb2025_sentry_bringup loopback_decision_sim.launch.py \
  use_rviz:=False \
  publish_decision_mode:=False
```

然后手动发布：

```bash
ros2 topic pub /decision/sim_mode std_msgs/msg/String "{data: anchor}" -1
```

## 8. 实车部署

### 8.1 先回答你的两个问题

1. 旧的 `robot_decision_node` 不应该再作为主决策入口继续启动。
2. 实车不要启动 `loopback_navigation.launch.py`，它是轻量仿真链路的一部分，不是实车导航入口。

当前工程里，推荐的实车总入口是：

- `src/pb2025_sentry_bringup/launch/bringup.launch.py`

这个入口已经会同时拉起：

- `standard_robot_pp_ros2` 串口与底层节点
- 实车导航入口 `rm_navigation_reality_launch.py`
- `pb2025_sentry_behavior` 行为树服务端与客户端

并且在这个入口里，传给 `standard_robot_pp_ros2.launch.py` 的参数已经是：

```python
"launch_robot_decision": "False"
```

也就是说：

- 如果你用 `bringup.launch.py` 启动，旧决策节点本来就不应该启动
- 不需要再去用 `loopback_navigation.launch.py` 顶替实车导航

### 8.2 现在代码中的实际状态

当前 `standard_robot_pp_ros2.launch.py` 里，旧 `robot_decision_node` 这一段本身就是注释状态。

所以目前结论是：

- 旧决策节点已经退出主链
- 实车主决策入口已经是 `pb2025_sentry_behavior`

### 8.3 实车推荐启动方式

最推荐直接使用一条总启动命令：

```bash
source install/setup.bash
ros2 launch pb2025_sentry_bringup bringup.launch.py \
  world:=rmul \
  use_rviz:=false
```

如果你要在实车端本机也打开 RViz，可以改成：

```bash
source install/setup.bash
ros2 launch pb2025_sentry_bringup bringup.launch.py \
  world:=rmul \
  use_rviz:=true
```

其中：

- `world:=rmul` 对应你实际使用的地图名
- `map` 和 `prior_pcd_file` 会按这个名字自动拼接
- 行为树参数默认使用 `sentry_behavior.yaml`

### 8.4 如果你想分开启动

虽然不推荐，但如果你确实要拆开启动，正确关系应该是：

1. 启动底层与串口：

```bash
source install/setup.bash
ros2 launch standard_robot_pp_ros2 standard_robot_pp_ros2.launch.py \
  launch_robot_decision:=False
```

2. 启动实车导航：

```bash
source install/setup.bash
ros2 launch pb2025_nav_bringup rm_navigation_reality_launch.py
```

3. 启动行为树：

```bash
source install/setup.bash
ros2 launch pb2025_sentry_behavior pb2025_sentry_behavior_launch.py \
  params_file:=/home/ats/ats_sentry_ws/install/pb2025_sentry_behavior/share/pb2025_sentry_behavior/params/sentry_behavior.yaml
```

但更建议你还是直接用：

```bash
ros2 launch pb2025_sentry_bringup bringup.launch.py ...
```

因为它已经把三条链路接好了。

### 8.5 实车和仿真的入口区别

你后面可以按下面这个原则记：

- 仿真总入口：`loopback_decision_sim.launch.py`
- 仿真导航入口：`loopback_navigation.launch.py`
- 实车总入口：`bringup.launch.py`
- 实车导航入口：`rm_navigation_reality_launch.py`

所以：

- `loopback_navigation.launch.py` 不能当实车行为树入口
- 实车行为树应该跟着 `bringup.launch.py` 一起启动

### 8.6 实车启动前检查项

建议你在实车部署前至少确认这几项：

1. 串口设备名、波特率和底层参数已经在 `node_params.yaml` 或串口配置中对应正确
2. `world` 对应的地图 yaml 和 pcd 文件真实存在
3. `sentry_behavior.yaml` 中的目标点、巡逻点、血量阈值都是你当前车的版本
4. 实车使用 `decision.input_source: referee`
5. 裁判系统话题、里程计、TF、激光数据都能正常发布

### 8.7 一句结论

如果你现在要上实车，正确理解是：

- 不要启动 `loopback_navigation.launch.py`
- 不要再让旧 `robot_decision_node` 参与主链
- 直接用 `bringup.launch.py` 作为实车总入口

## 10. 后续调参建议

如果你后面追求更高的平滑性，建议按下面顺序调：

1. 先把 `NavigateThroughPoses` 继续保留为主执行方式
2. 增加 `decision.point_roles.patrol_indices` 的点数
3. 把 `decision.decision_config.patrol_preview_points` 调到 3 或 4
4. 再去调 Nav2 的 planner、controller、smoother 参数

其中第 2 和第 3 步最直接。

如果只有 2 个巡逻点，就算已经切到 `NavigateThroughPoses`，丝滑程度也还是有限。
