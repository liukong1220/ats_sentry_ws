# 哨兵导航与决策移植说明

## 1. 当前结论

当前工程已经完成以下迁移结果：

- `rmul_2026.xml` 是新的主决策树
- 决策执行方式已经切换为 `NavigateThroughPoses`
- 默认轻量仿真不再依赖裁判系统时间、血量、RFID
- loopback 默认走 `simulation` 输入源
- 同时保留了可选的“手动裁判仿真输入”模式，便于测试比赛状态、血量、时间分支

这意味着你现在的项目已经不是 `goal_pose` 方案，而是：

- 决策层：`pb2025_sentry_behavior`
- 导航执行层：`Nav2 NavigateThroughPoses`
- 轻量仿真层：`nav2_loopback_sim`

## 2. 目录与入口

## 3. 现在的决策结构

### 3.1 顶层输入源

`rmul_2026` 现在支持两种输入源：

- `decision.input_source: simulation`
- `decision.input_source: referee`

行为树会先判断输入源，再选择对应分支：

- `simulation` 分支只看 `decision/sim_mode`
- `referee` 分支继续看比赛状态、剩余时间、当前血量

### 3.2 simulation 分支

支持四种仿真模式：

- `patrol`
- `anchor`
- `retreat`
- `safe`

对应关系：

- `patrol`：多点巡逻
- `anchor`：固定锚点驻守
- `retreat`：退防点
- `safe`：安全点

### 3.3 referee 分支

优先级从高到低：

1. 危险血量 `critical` -> 安全点
2. 低血量 `low` -> 退防点
3. 关键时间 `critical` 且血量正常 -> 关键时间固定点
4. 正常血量 -> 巡逻

## 4. 现在为什么更平滑

这次移植里，决策输出已经不是反复发布单个 `/goal_pose`，而是直接调用 `NavigateThroughPoses`：

- 巡逻路径可以一次性交给 Nav2 连续执行
- 分支不变时，不会每个 tick 重复取消并重发同一路径
- 分支切换时，会主动取消旧路径并派发新路径

同时增加了参数：

- `decision.decision_config.patrol_preview_points`

这个参数的作用是：

- 决定一次送给 `NavigateThroughPoses` 的巡逻预览点数量
- 数值越大，巡逻点足够多时越连续
- 不需要重新编译，只要改 YAML

注意：

- 如果你的 `patrol_indices` 只有 2 个点，那么 `patrol_preview_points` 调再大，提升也有限
- 真正想更丝滑，应该同时增加巡逻点数量和路径密度

运行中临时调这个参数也可以：

```bash
ros2 param set /pb2025_sentry_behavior_server decision.decision_config.patrol_preview_points 3
```

如果你是改 YAML，则重新启动 launch 即可，不需要重新编译。

## 6. 轻量 loopback 默认运行方式

默认启动：

```bash
source install/setup.bash
ros2 launch pb2025_sentry_bringup loopback_decision_sim.launch.py
```

如果你手动指定，也推荐写成：

```bash
ros2 launch pb2025_sentry_bringup loopback_decision_sim.launch.py use_rviz:=true
```

默认行为：

- 使用 `sentry_behavior_loopback.yaml`
- `decision.input_source=simulation`
- `fake_decision_sim_inputs.py` 发布 `decision/sim_mode`
- 行为树执行 `rmul_2026`

运行中切换模式：

```bash
ros2 param set /fake_decision_sim_inputs decision_mode patrol
ros2 param set /fake_decision_sim_inputs decision_mode anchor
ros2 param set /fake_decision_sim_inputs decision_mode retreat
ros2 param set /fake_decision_sim_inputs decision_mode safe
```

如果想完全自己手动发仿真模式 topic：

```bash
ros2 launch pb2025_sentry_bringup loopback_decision_sim.launch.py \
  use_rviz:=False \
  publish_decision_mode:=False
```

然后手动发布：

```bash
ros2 topic pub /decision/sim_mode std_msgs/msg/String "{data: anchor}" -1
```

## 7. 手动裁判仿真输入模式

如果你想在仿真里测试“比赛状态 + 时间 + 血量”分支：

```bash
source install/setup.bash
ros2 launch pb2025_sentry_bringup loopback_decision_sim.launch.py \
  use_rviz:=true \
  publish_referee_inputs:=True \
  publish_decision_mode:=False \
  behavior_params_file:=/home/ats/ats_sentry_ws/install/pb2025_sentry_behavior/share/pb2025_sentry_behavior/params/sentry_behavior.yaml
```

这个模式下：

- 行为树走 `referee` 输入源
- `fake_decision_sim_inputs.py` 周期发布裁判话题
- 你可以实时改参数测试分支

实时修改血量：

```bash
ros2 param set /fake_decision_sim_inputs current_hp 100
```

实时修改比赛剩余时间：

```bash
ros2 param set /fake_decision_sim_inputs stage_remain_time 30
```

实时修改比赛阶段：

```bash
ros2 param set /fake_decision_sim_inputs game_progress 4
```

如果你想完全自己手动发布裁判话题，也可以：

```bash
ros2 launch pb2025_sentry_bringup loopback_decision_sim.launch.py \
  use_rviz:=False \
  publish_referee_inputs:=False \
  publish_decision_mode:=False \
  behavior_params_file:=/home/ats/ats_sentry_ws/install/pb2025_sentry_behavior/share/pb2025_sentry_behavior/params/sentry_behavior.yaml
```

然后自行向这些话题发布：

- `/referee/game_status`
- `/referee/robot_status`
- `/referee/rfid_status`

## 9. 后续调参建议

如果你后面追求更高的平滑性，建议按下面顺序调：

1. 先把 `NavigateThroughPoses` 继续保留为主执行方式
2. 增加 `decision.point_roles.patrol_indices` 的点数
3. 把 `decision.decision_config.patrol_preview_points` 调到 3 或 4
4. 再去调 Nav2 的 planner、controller、smoother 参数

其中第 2 和第 3 步最直接。

如果只有 2 个巡逻点，就算已经切到 `NavigateThroughPoses`，丝滑程度也还是有限。
