## 4. 现在的整体架构

### 4.1 仿真链路

Loopback 仿真模式下，数据流是这样的：

1. `loopback_simulator` 发布 `/clock`、`/odom`、`/scan`、TF。
2. `fake_decision_sim_inputs.py` 发布假的裁判系统输入和 `initialpose`。
3. `pb2025_sentry_behavior_server` 订阅裁判信息、`/odom`、代价地图，并把它们写进行为树黑板。
4. `pb2025_sentry_behavior_client` 请求执行 `rmul_2026`。
5. `rmul_2026.xml` 根据时间阶段和血量区间选择目标路径。
6. `PublishDecisionGoal` 把最终目标点发布到 `/goal_pose`。
7. `bt_navigator` 订阅 `/goal_pose`，触发 Nav2 导航。
8. `controller_server` 输出 `/cmd_vel`，loopback 仿真器再根据速度更新位姿。

所以 loopback 模式下不需要：

- 实机串口
- 实机雷达
- 实机底盘
- Gazebo

### 4.2 实机链路

真实模式下，主链路是：

1. `standard_robot_pp_ros2.launch.py` 启动串口、底层接口等节点。
2. 其中旧 `robot_decision_node` 已在 `bringup.launch.py` 中被显式关闭。
3. `pb2025_sentry_behavior` 作为新的决策入口独立启动。
4. Nav2 继续负责路径规划与控制。
5. 行为树通过 `/goal_pose` 把目标点发给 Nav2。

也就是说，实机模式仍然可以使用串口和真实硬件，但“决策是谁来发布导航目标”这件事已经从旧节点切换到了行为树。

## 5. `rmul_2025` 与 `rmul_2026` 的区别

### 5.1 `rmul_2025`

- 保留旧版
- 仍然是简单规则树
- 只适合作为旧逻辑保留和回退参考
- 不承载本次迁移的多分支决策逻辑

### 5.2 `rmul_2026`

- 是本次迁移后的新主树
- 使用行为树规范表达旧决策分支
- 使用 topic 风格发布 `/goal_pose`
- 依赖当前位姿 `decision_current_pose` 判断“目标是否到达”
- 去掉了 `stop_flag` 主逻辑
- 更适合后续继续扩展成真正的导航决策树

## 6. `rmul_2026.xml` 的决策逻辑

`rmul_2026` 顶层是一个持续运行的 `ReactiveSequence`，所以每一轮 tick 都会重新检查当前比赛状态和血量。

它的优先级从高到低如下：

1. `critical hp`
   进入安全点分支。
2. `low hp`
   进入低血量退防分支。
3. `critical time + normal hp`
   进入关键时间固定点分支。
4. `normal hp`
   进入巡逻分支。

### 6.1 危险血量分支

触发条件：

- `IsHpBand band="critical"`

动作：

- 重置低血量粘滞目标
- 选中 `decision_supply_safe_point_index`
- 生成到安全点的路径
- 如果当前已经到达该路径终点，就保持当前状态
- 如果还没到，就提取路径最后一个点并发布到 `/goal_pose`

### 6.2 低血量分支

触发条件：

- `IsHpBand band="low"`

动作：

- 根据当前位置，从 `decision.point_roles.low_hp_candidate_indices` 中选最近退防点
- 这个目标会写入 `decision_low_hp_target_index`
- 后续 tick 会优先保持这个退防目标，避免低血量时来回抖动
- 到达后再通过 `ResetLowHpTarget` 清空粘滞目标

### 6.3 关键时间分支

触发条件：

- `IsGameTimeStage stage="critical"`
- `IsHpBand band="normal"`

动作：

- 选中 `decision_critical_time_target_index`
- 发布固定关键点目标

### 6.4 正常巡逻分支

触发条件：

- `IsHpBand band="normal"`

动作：

- 根据 `decision_patrol_cursor` 和 `decision_patrol_direction` 选当前巡逻段
- 如果路径终点到达，则推进到下一巡逻状态
- 如果还没到达，则发布当前巡逻段终点到 `/goal_pose`

这里使用的是“往返巡逻”逻辑，不是环形一圈一圈跳转。

## 7. 新增行为树节点作用

### 7.1 条件节点

- `IsGameTimeStage`
  根据 `referee/game_status.stage_remain_time` 判断比赛时间阶段。
- `IsHpBand`
  根据比赛剩余时间和血量阈值判断当前血量区间。
- `IsPathGoalReached`
  用当前里程计位姿判断是否已经到达当前路径末端。

### 7.2 动作节点

- `SelectFixedPath`
  从固定索引选出路径。
- `SelectNearestRetreatPath`
  从多个退防候选点里选最近点。
- `SelectPatrolPath`
  生成巡逻路径，并给出下一巡逻游标。
- `AdvancePatrolCursor`
  把下一巡逻状态写回黑板。
- `ResetLowHpTarget`
  清空低血量粘滞目标。
- `SelectPathGoalPose`
  从当前路径中取最后一个 pose 作为导航目标。
- `PublishDecisionGoal`
  把目标发布到 `/goal_pose`。

## 8. 黑板里的关键变量

- `decision_current_pose`
  当前位姿，由 `/odom` 或 `/odometry` 写入。
- `decision_path`
  当前分支选出的路径。
- `decision_goal_pose`
  当前要发给 Nav2 的最终目标点。
- `decision_patrol_cursor`
  当前巡逻点游标。
- `decision_patrol_direction`
  当前巡逻方向。
- `decision_next_patrol_cursor`
  下一个巡逻游标。
- `decision_next_patrol_direction`
  下一个巡逻方向。
- `decision_low_hp_target_index`
  低血量时粘滞保存的退防点索引。
- `decision_supply_safe_point_index`
  危险血量时的安全点索引。
- `decision_critical_time_target_index`
  关键时间阶段的固定目标点索引。

## 10. 如何切换行为树

如果你要切树，改：

`src/pb2025_sentry_behavior/params/sentry_behavior.yaml`

具体字段：

```yaml
pb2025_sentry_behavior_client:
  ros__parameters:
    target_tree: rmul_2026
```

如果你想切回旧树，只需要改成：

```yaml
target_tree: rmul_2025
```

## 14. `PublishDecisionGoal` 的当前策略

新的 topic 风格执行节点现在采用下面的发布策略：

1. publisher 在节点初始化时就提前创建。
2. 目标发生变化时立即发布。
3. 为避免首包丢失，会在很短时间内补发少量同目标消息。
4. 不会对同一个目标做固定周期重发。

这样做的原因是：

- topic 风格比 action 风格更稳定，规避了此前 `SendNavThroughPoses` 在 halt 阶段的异常
- 提前创建 publisher，可以把 ROS 图发现过程放到第一次真正发目标之前
- 少量短时补发可以降低首包丢失概率
- 避免对同一个目标持续重发，从而触发 `bt_navigator` 长时间反复 preemption，影响平滑性

## 15. 为什么 `rmul_2026` 不再使用 `stop_flag`

旧决策里 `stop_flag` 更像一种人工插入的导航停顿控制。

在当前迁移版本中去掉它，主要是为了：

- 减少分支切换时的不必要停顿
- 提高多点巡逻与状态切换的连续性
- 避免对响应速度和平滑性造成额外影响

当前项目里仍然可能保留 `stop_flag` 相关的旧节点或话题兼容能力，但 `rmul_2026.xml` 的主决策逻辑已经不依赖它。

## 17. 当前仍然建议你注意的点

### 17.1 ROS 2 脏环境

如果你重复 launch 很多次，可能遇到 FastDDS 残留问题，表现为：

- 图里看得到节点，但服务请求发不进去
- action 看得到，但 `send_goal` 卡住
- 日志里出现 `Failed init_port fastrtps_portxxxx`

出现这种情况，先做清理：

```bash
ros2 daemon stop
pkill -f pb2025_sentry_behavior
pkill -f loopback_simulator
pkill -f lifecycle_manager
rm -f /dev/shm/fastrtps*
rm -f /dev/shm/sem.fastrtps*
```

然后重新 `source install/setup.bash` 再启动。

### 17.2 行为树参数与导航参数分离

现在建议你保持下面这个原则：

- 决策树参数只改 `sentry_behavior.yaml`
- 导航参数只改 `node_params.yaml`

不要再把两者重新混回一个文件里，不然后面很容易出现“我改了参数但实际没生效”的错觉。

## 18. 一句总结

现在这个项目已经是一个“导航 + 行为树决策 + loopback 仿真”一体化框架：

- 仿真时不依赖实机
- 实机时旧决策节点已退出主链
- 新决策树 `rmul_2026` 已能闭环驱动 Nav2
- 老树 `rmul_2025` 被保留作回退参考
